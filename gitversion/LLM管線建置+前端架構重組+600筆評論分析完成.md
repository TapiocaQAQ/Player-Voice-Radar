### `backend/llm_service.py`（新增，310 行）

Groq LLM 批次分析管線，Step 2 資料處理管線。核心功能與歷經的問題修復如下：

#### 核心邏輯

| 功能 | 說明 |
|------|------|
| 增量更新 | 讀取 `cached_reviews.json`，以 `review_id` Set 去重，只送未處理的新評論 |
| 批次處理 | `BATCH_SIZE = 3`，每批動態計算 `max_tokens = max(1024, batch_size × 500)` |
| 輸出合併 | 新結果 + 既有快取合併，依 `date` 降冪排序後寫回 |
| `--limit N` | 先切 `all_reviews[:N]` 再區分已快取/新評論，終端機清楚顯示跳過與待處理筆數 |
| `--test` | 強制送前 5 筆驗證格式，不寫入快取 |

#### 問題一：JSON 截斷（`Expecting ',' delimiter`）

**根因**：`qwen/qwen3-32b` 是 Thinking Model，會先輸出 `<think>...</think>` 推理鏈（消耗 1,000–2,000 tokens），再輸出 JSON。Groq 的 6,000 TPM 上限計算整個請求（input + max_tokens），原本 `BATCH_SIZE = 10` + `max_tokens = 3,500` 共 ~5,000 tokens，Thinking 吃完後剩不到 500 tokens，根本不夠寫完 10 筆 JSON。

**解決**：`BATCH_SIZE` 從 10 → 5 → 最終 **3**，即使 Thinking 吃掉 1,500 tokens，仍剩 ~1,500 tokens 足夠輸出 3 筆 JSON。

#### 問題二：Rate Limit 429 資料遺失

**根因**：原始程式在 429 時直接跳過，整批資料永久遺失。

**解決**：從錯誤訊息自動 parse `try again in X.XXs`，sleep 後重試同一批，不丟資料：
```python
if "rate_limit_exceeded" in err_str or "429" in err_str:
    wait_match = re.search(r"try again in (\d+\.?\d*)s", err_str)
    wait = float(wait_match.group(1)) + 2.0 if wait_match else RETRY_DELAY
    time.sleep(wait)
    # 重試同一批
```

#### 問題三：無重試機制

**解決**：`process_batch()` 加入最多 3 次重試（`MAX_RETRIES = 3`）：
- `json.JSONDecodeError`：sleep 2s 後重試
- Rate Limit 429：解析建議等待秒數 + 2s 緩衝後重試
- 其他錯誤：直接往上拋

#### 問題四：批次全失敗的 Fallback 機制

**設計需求**：「500 筆就是 500 筆，不要重新跑才抓到。」

**解決**：批次 3 次重試全失敗 → 自動降為逐筆（batch of 1）送出 → 逐筆仍失敗 → 寫入 `backend/bad_data.json`（含去重保護）。

```
批次 3 筆 → 失敗 3 次
    ↓ [FALLBACK]
    ├─ 第 1 筆 → 單獨送 → 成功
    ├─ 第 2 筆 → 單獨送 → 成功
    └─ 第 3 筆 → 單獨送 → 失敗 → 寫入 bad_data.json
```

#### 問題五：Windows GBK 編碼崩潰

**根因**：Windows 終端機預設 GBK encoding，print 含 emoji（🎉、🔍）導致 `UnicodeEncodeError`。

**解決**：全面改用純 ASCII 標籤（`[DONE]`、`[INFO]`、`[RETRY]`、`[RATE LIMIT]`、`[FALLBACK]`）。

#### 最終參數設定

| 參數 | 初始值 | 最終值 | 原因 |
|------|--------|--------|------|
| `BATCH_SIZE` | 10 | **3** | 避免 Thinking 吃掉輸出 token 空間 |
| `max_tokens` | 固定 8,192 | **`max(1024, batch×500)`** | 動態計算，避免超過 6,000 TPM |
| `MAX_RETRIES` | 0 | **3** | JSON 截斷偶發，重試可解決 |
| Rate Limit 處理 | 直接跳過 | **解析秒數 + sleep + 重試** | 不丟資料 |
| 批次全失敗 | 跳過丟失 | **逐筆 fallback → bad_data.json** | 確保目標筆數 100% 達成 |
